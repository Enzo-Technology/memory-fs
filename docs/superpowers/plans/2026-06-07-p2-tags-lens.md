# Tags Lens (+ list-by-tag backend) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the P2 **Tags lens** to the memory browser. The store already exposes the tag *vocabulary* (`browse?kind=tags` → `TagItem{tag,count}`); what is missing is "given a tag, list its memories." This plan adds a `kind=tagged` backend query (store + browse-api, additive), then wires a Tags lens in the UI: the lens shows the tag vocabulary as clickable rows; picking a tag lists that tag's memories (reusing the existing flat-row path), with a "← all tags" back row to return to the vocabulary.

**Architecture:** Backend is SQLite (better-sqlite3) behind `MemoryStore` (`src/core/store.ts`), exposed read-only over a session-auth JSON API (`src/lib/browse-api.ts`). The UI (`ui/src/`) is a React SPA: `useBrowser` is the deep state module, the panes (`TreePane`, `LensRow`, `Browser`) are thin over its view-model, and `api.ts` is the only place that knows endpoint URLs (response types imported type-only from the server's store types). The Tags lens follows the existing two-level pattern (`namespaces` lens = vocabulary → leaves): vocabulary first, drill into one tag's memories second.

**Tech Stack:** TypeScript, better-sqlite3 + FTS5, Node `http`, React. Backend typechecked by `tsc` (in `npm run build`) and tested by Vitest (`tests/**/*.test.ts`). UI bundled by **Vite** (`npm run build:ui`).

  ```ts
    // List the memories carrying a given tag, newest first. The tag *vocabulary* lives in
    // tagVocabulary (kind=tags); this is the drill-in: pick a tag, see its memories. Records, so
    // it carries a snippet (ADR 0002), matching recent/hubs/orphans.
    private tagged(tag: string, limit: number): RecentItem[] {
      const rows = this.db
        .prepare<unknown[], Omit<RecentItem, "snippet"> & { content: string }>(
          `SELECT m.namespace, m.key, m.type, m.updated_at, m.content FROM memories m
           WHERE EXISTS (SELECT 1 FROM tags t WHERE t.memory_id = m.id AND t.tag = @tag)
           ORDER BY m.updated_at DESC
           LIMIT @limit`,
        )
        .all({ tag, limit });
      return rows.map(({ content, ...item }) => ({ ...item, snippet: snippet(content) }));
    }
  ```

## Self-review notes

- **Backend is genuinely additive.** `tagged` is a new `BrowseKind` arm, a new private method, and a new optional `tag` param — no existing `recent`/`hubs`/`orphans`/`tags`/`namespaces` behavior changes. The `tags` (vocabulary) route is explicitly re-tested in Task 3 to prove no regression.
- **Snippet invariant honored.** `tagged` returns records, so it carries a snippet via the exact `{ content, ...item } => ({ ...item, snippet: snippet(content) })` mapping the other record-returning queries use. The existing ADR-0002 fitness test in `tests/store.test.ts` is left intact and still passes.
- **Boundary guard, not defensive sprawl.** A `kind=tagged` request with no `tag` returns empty items (decided in the spec). This is validated at the one boundary (`browse()`); the private `tagged(tag, ...)` trusts it receives a tag. No "just in case" fallbacks elsewhere.
- **Two-level lens reuses the established pattern.** Tags mirrors the `namespaces` lens (vocabulary → drill), and the drilled list reuses the existing `flat` rows + `toRows` + `MemoryRow` path rather than adding a component — only `toRows` gains the `tagged` arm (same `RecentItem` shape as `recent`). The flat-lens effect is guarded `lens==="tags" && !selectedTag` exactly as `namespaces` is guarded.
- **Resolved ambiguity — bundler.** The task brief said the UI is bundled by esbuild and "don't run tsc -p ui." The repo actually bundles the UI with **Vite** (`ui/vite.config.ts`, `npm run build:ui` → `vite build ui`). The plan uses `npm run build:ui` (correct either way) and still avoids a standalone UI typecheck. Note: Vite/Rollup does NOT typecheck the UI, so a wrong type in `useBrowser.ts`/`TreePane.tsx` may build clean — the type-only imports from `../../src/core/store` are the safety net, and `npm run build` (which runs `tsc` over the backend) catches any store-side type error.
- **Resolved ambiguity — `total`.** `browse()` computes `total` as the store-wide memory COUNT (not the tagged count). The `tagged` arm follows suit (`total` = whole store), consistent with every other arm; the running-total UI already treats `total` as store-wide. The number of tagged memories is simply `items.length` (capped at limit 100), surfaced implicitly by the row list, not as a separate count — matching how the namespace-leaf drill behaves.
- **Resolved ambiguity — an "all"/"tags" lens already exists partially.** The brief's "add tags lens" wording implied a greenfield lens, but the UI already ships an `all` lens and the `Lens` union already has `"all"`. The plan inserts `"tags"` into the existing union/`LENSES` array (after Orphans, per spec) without disturbing `all`.
- **`selectTag` reset.** `selectLens` now clears `selectedTag` so switching away from Tags (or back to it) starts at the vocabulary, and the `selectedTag` dependency was added to both the flat-lens and tags-vocabulary effects so the two states swap cleanly. `onClearTag` is wired as `() => vm.selectTag(null)` in `Browser.tsx` (the brief left the exact prop name open; `onSelectTag`/`onClearTag` chosen for symmetry).
- **Out of scope (not added):** no tag count badge in the drilled header, no tag filtering combined with namespace, no URL/route persistence of the selected tag (the address-as-route covers memories, not lens state — consistent with the other lenses). Flag if Ben wants tag selection to survive reload.
