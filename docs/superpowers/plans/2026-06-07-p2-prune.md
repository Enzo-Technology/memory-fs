# Prune (DELETE + backlink guardrail) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **⚠ DESTRUCTIVE:** this batch deletes data. Execute it LAST, after the other P2 plans. Every delete path is session-gated, backlink-guardrailed, force-flagged, and confirmed.

**Goal:** Add the one mutation in the memory browser — delete a memory — from the Reader pane. A `DELETE /api/memories/:ns/:key` endpoint (session-gated) calls the existing `store.del(ns,key,force)`. The UI confirms before deleting and, when inbound backlinks exist, refuses with a guardrail panel that lists the linking memories and requires an explicit "Delete anyway".

**Architecture:** The store already owns the destructive logic and the guardrail (`del` throws when `backlinkCount>0 && !force`; `backlinks` lists inbound records). This plan adds: (1) a DELETE branch to the read-only `browse-api.ts` request handler, sitting after the existing 401 session guard; (2) `deleteMemory` in the UI data layer (`api.ts`); (3) delete orchestration + guardrail state in `useBrowser.ts`; (4) presentational delete/confirm/guardrail controls in `Reader.tsx` + CSS. No new modules.

**Tech Stack:** TypeScript, Node `http`, better-sqlite3 (backend); React + esbuild (UI); Vitest (`tests/**/*.test.ts`).

- **Test patterns:** backend tests open a fresh better-sqlite3 store via `freshStore()` and use the `fakeRes` / `withSession` / `noSession` / `call` helpers in `tests/browse-api.test.ts`.       const url = new URL(req.url ?? "/", "http://localhost");
  ```ts
      const path = url.pathname;
      const q = url.searchParams;

      // DELETE /api/memories/:namespace/:key — the one write. Session-gated above. The store
      // refuses when inbound backlinks exist unless force=true; we pre-check so we can return
      // the linking records (409) for the UI's guardrail instead of an opaque throw.
      if (req.method === "DELETE") {
        const m = path.match(/^\/api\/memories\/([^/]+)\/([^/]+)$/);
        if (!m) return json(res, 404, { error: "not found" });
        const namespace = decodeURIComponent(m[1]!);
        const key = decodeURIComponent(m[2]!);
        const force = q.get("force") === "true";
        if (!force) {
          const bl = store.backlinks(namespace, key);
          if (bl.length > 0) {
            return json(res, 409, { error: "has backlinks", backlinks: bl });
          }
        }
        const ok = store.del(namespace, key, force);
        if (!ok) return json(res, 404, { error: "not found" });
        return json(res, 200, { ok: true });
      }

      // GET /api/memories/recall — full-text search. Checked before the detail pattern below;
  ```

  ```ts
  // The one write: prune a memory. 200 → gone; 409 → the store's backlink guardrail tripped, so
  // we hand back the linking records for the UI to surface before a forced retry. Mirrors `get`'s
  // 401→sign-in bounce. credentials:"include" carries the session cookie.
  export async function deleteMemory(
    namespace: string,
    key: string,
    force = false,
  ): Promise<{ ok: true } | { conflict: true; backlinks: Backlink[] }> {
    const path =
      `/api/memories/${encodeURIComponent(namespace)}/${encodeURIComponent(key)}` +
      (force ? "?force=true" : "");
    const res = await fetch(path, { method: "DELETE", credentials: "include" });
    if (res.status === 401) {
      location.href = "/sign-in";
      throw new Error("unauthenticated");
    }
    if (res.status === 409) {
      const body = (await res.json()) as { backlinks: Backlink[] };
      return { conflict: true, backlinks: body.backlinks };
    }
    if (!res.ok) throw new Error(`${res.status}: ${await res.text()}`);
    return { ok: true };
  }
  ```

## Decisions / deviations

- **DESTRUCTIVE — execute LAST.** This is the only mutation in the browser and it permanently deletes rows. It must be the final P2 batch, after the read-only P2 work (All lens, chevron fix, login redesign, Tags, ⌘K, wikilinks) is merged. The conventions section and header both flag this.
- **No optimistic-undo (deliberate deviation from the spec).** The spec's prune section said "optimistic removal, undo toast" for the no-backlink path. Undo would require RE-CREATING the deleted memory, but the browser has NO create/authoring path — authoring is explicitly out of scope (spec §"Out of scope" and §Boundaries). An undo toast that cannot actually restore the record is a worse lie than no undo. This plan instead uses **confirm-then-delete with no undo** — a strictly stronger guard — plus the backlink guardrail. If undo is ever wanted, it needs an authoring endpoint first; that is a separate, out-of-scope decision.
- **Backend pre-check vs. store throw.** The API pre-checks `backlinks()` so it can return the linking records (409) for the UI, rather than catching the store's opaque throw. The store throw is kept as the deeper invariant (Task 3 gate). This is intentional redundancy on a destructive path.
- **Reader is the only prune surface (Orphans-lens delete deferred).** The spec mentioned pruning from the Orphans lens. To keep this destructive batch small and single-surfaced, delete lives only in the Reader (open the memory, then delete it). A per-row delete affordance on Orphans rows is a follow-up, noted here, not built — the Reader delete fully satisfies the "prune" verb.
- **Confirm is the gate that replaces a modal.** No separate confirm dialog/modal is introduced; the inline two-step (Delete → Delete/Cancel) in `reader__bar` is the explicit confirmation, consistent with the lines-and-space Foundations system.
