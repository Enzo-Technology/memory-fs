Does the name of a tool change what a model files in it?

Tool-wording pilot · 2026-06-07, corrected re-run 2026-06-08 · memory-fs · 2 models, n=5 per cell · cross-checked on two independent agent runners

We're building a shared memory store that agents reach over MCP. The open product question was small and stubborn: do we call it memory, context, or knowledge? It feels like branding. We suspected it wasn't.

The bet: a tool's name + description + result strings are its filing policy. They're the only thing telling a model what belongs in the store, when to read it, and when to clean it up. If models carry the everyday connotations of these words — memory = personal, episodic, automatic; context = curated, shared, load-before-you-act; knowledge = settled reference — then the same backend under different vocabularies should produce measurably different behavior. So this is an interface question wearing a branding costume.

How we tested it

One renameable MCP server sits over the real store. A single env var swaps the surface: the noun changes (memory_ / context_ / knowledge_) while the verbs stay neutral and identical (_write, _read, _search, _browse, …) and the argument schemas never move. The only signal that differs between conditions is the vocabulary — the noun and the description prose. Backend, data, and tool arguments are byte-for-byte the same.

The model never gets a hint. The system prompt says nothing about a store; the user turns never name one. Whatever the model does — read first, write trivia, file a decision — comes entirely from reading the tool surface. Probes:

  • Cold start — "draft a headline" (does it read the store before acting?), then a direct recall question.
  • Working session, dropped on top of a real scrubbed coding chat — a contradiction ("we're dropping the desktop app"), a personal aside ("my dog Rex was barking at the mailman"), and a durable decision ("all errors go through AppError from now on").
  • Wrap-up — an explicit "tidy up before we close out."

Two same-tier models: Claude Sonnet 4.6 and OpenAI gpt-5.5, both reasoning at low depth. Every cell ran n=5. Metrics are deterministic, read straight off the tool-call logs.

(Result strings matched too — a memory write replies "Memory saved."; a context write replies "Shared context updated.")

Round 1 — memory vs context

The pilot ran just two conditions, M and C, to see if there was any separation worth chasing. There was, and it was blunt.

The cleanest result is the dog. On the "my dog Rex" aside — pure personal trivia, junk for a shared team store — memory invited a write almost every time and context suppressed it completely.

Trivia write (the dog aside)memory (M)context (C)
Sonnet 4.65 / 50 / 5
gpt-5.53 / 50 / 5

The other half of round 1: context's "read before acting" framing made gpt-5.5 read the store first every single time (5/5 under context vs 0/5 under memory). Sonnet read before acting in both conditions — in fact it reads in every condition we tested. So the provider difference is real, but it isn't "one reads, one doesn't": Sonnet always reads; gpt-5.5 reads only when the description tells it to. The wording effect on reading is visible in the model that isn't saturated.

Round 1 verdict: wording moves behavior, and it moves it exactly where it should — on discriminating junk from signal. "Memory" says keep things about the user, so the model keeps the dog. "Context" says shared team record, so the dog gets dropped. Worth a second round.

Round 2 — knowledge, and the crossed conditions

Round 2 added knowledge (K) plus the two experiments that actually answer the product question: the crossed conditions. MxC is a memory name with a context description; CxM is a context name with a memory description. If the noun is what matters, behavior follows the name. If the description is what matters, behavior follows the prose — regardless of what the tool is called.

A second, independent runner agrees

Because the read metric had been wrong once already (see the correction below), we re-ran the cold-start probe through a different agent runner — MCPJam's SDK (Host + MCPClientManager, built on the Vercel AI SDK) — against the same renameable server, same seeded store, same neutral system prompt. Different loop, different tool plumbing, our scoring untouched. If the corrected story is real and not an artifact of our own harness, two runners should land in the same place. They do:

Cold-start read (m1 / T1)our runnerMCPJam runner
Sonnet — every condition5 / 55 / 5
gpt-5.5 · M (memory)0 / 50 / 5
gpt-5.5 · CxM (memory description)2 / 50 / 5
gpt-5.5 · C (context)5 / 55 / 5
gpt-5.5 · MxC (context description)5 / 55 / 5
gpt-5.5 · K (knowledge)5 / 53 / 5

Both runners: Sonnet reads in 100% of cells; gpt-5.5 is suppressed by the memory framing (M, CxM) and triggered by the context/knowledge framing (C, MxC, K). The two partial cells (CxM, K) wobble a few runs apart — expected at n=5 with the runners' differing internals — but the direction is identical. The "Sonnet always reads" result in particular reproduces perfectly, which is exactly the claim the old harness got wrong.

Correction (post-hoc). An earlier version of this writeup reported that Sonnet "never reads unprompted." That was a harness bug, not a model behavior: our agent loop counted a same-message text preamble ("Let me check the store first…") as "answered before reading," which biased the read metric against models that narrate before calling a tool (Anthropic does; OpenAI doesn't). We fixed it by scoring reads from the tool-call log itself, re-ran the whole dataset on the corrected harness, and cross-checked on a second independent runner (above). The trivia (m4) and duplicate-write (m3b) findings were unaffected — they're scored from write content, not turn ordering.
The headline: the surface is interface, not branding — but it's the description that does the work, not the noun. A tool called memory_ with a "shared team context, read before acting" description behaves like context. The word on the tin barely registers; the usage guidance underneath it is the whole lever. knowledge tracked context throughout — its "settled, durable" framing raises the same bar on junk.

The thing wording didn't fix

One probe broke the same way everywhere. When new information contradicted an existing record ("we're dropping the desktop app" vs. a stored record that says it ships with one), the right move is to update or supersede that record. Almost nobody did.

On contradiction (all conditions, both models)count
Updated / superseded the existing record (correct)3 / 50
Wrote a brand-new duplicate instead47 / 50
Wording can't save you here. Across every vocabulary and both models, the default is to append a near-duplicate and pollute the shared store — and to rarely even read the record it's about to contradict. This is a mechanism problem, not a naming one. It's the direct motivation for similarity-gated writes: no near-duplicate → write and mark confirmed; near-duplicate found → force a confirm round-trip so "update instead of duplicate" becomes the easy path. proposal: docs/proposals/2026-06-07-similarity-gated-writes.md

The rest saturated and washed out wording entirely: a genuine durable decision ("all errors go through AppError") got filed 100% of the time everywhere, and when explicitly told to "tidy up," everyone pruned. Wording matters for the ambiguous calls — junk-vs-signal, read-or-don't — not for the obvious ones.

So what do we call it?

The noun is nearly free to choose — pick whatever reads best for the product ("memory" is the most legible to users). What we cannot be casual about is the description: that's the filing policy. Write it like one. "Shared context the team relies on; read before acting; update when direction changes" produced the behavior we actually want — and it produced it even under the "memory" name.

Honest caveats

n=5 per cell — a screening run, not a powered study; treat magnitudes as provisional, though the clean splits (the dog, the cold-start read) would likely survive larger n. Single fixed phrasing per probe. Two providers, both at low reasoning depth. The duplicate-writes finding (47/50 appended a duplicate) is the most actionable result and the least about wording. These numbers are from a full re-run on the corrected read-metric harness (see the correction note above) and the cold-start read was independently reproduced on a second runner (MCPJam). Full numbers with Wilson confidence intervals live in RESULTS.md, regenerated deterministically from eval/artifacts/scores.json; the MCPJam cross-check lives in eval/artifacts-mcpjam/.

Appendix — what the models actually did

These are verbatim from the run logs (eval/artifacts/), lightly trimmed. The point of the experiment is abstract; the behavior is not.

The dog — same prompt, two surfaces

User turn (identical in both): "ha, sorry, got distracted — my dog Rex was barking at the mailman. anyway, where were we?"

Under memory_ (Sonnet) — it files the dog:

memory_write({ content: "User has a dog named Rex.", key: "user-pet", type: "user" }) // → "Memory saved." — junk, now permanent in a shared team store

Under context_ (Sonnet), identical backend — zero writes, it just answers and moves on:

// no tool calls reply: "No worries, hope Rex has calmed down! 😄 We were just wrapping up the folder structure for your data porting tool, with the key update being the switch to a CLI-first approach…"

Cold start — does it read before acting?

User turn: "Draft a one-line headline for the Acme landing page." The seeded store has positioning records that change the right answer.

Under context_ (gpt-5.5) — reads first, twice, before drafting:

context_search({ query: "Acme landing page headline brand messaging", limit: 5 }) context_search({ query: "Acme positioning tagline", limit: 5 }) // readBeforeAnswer: true

Under memory_ (gpt-5.5), same store — drafts blind:

// no tool calls — readBeforeAnswer: false

Under memory_ (Sonnet), same store — reads anyway, the same memory-named surface that gpt-5.5 drafts blind on. This is why "Sonnet never reads" was never true: it reads in every condition, so the wording can't move it.

memory_search({ query: "Acme team product description mission" }) memory_browse({ kind: "index", namespace: "acme" }) memory_browse({ kind: "index", namespace: "project:acme" }) memory_read({ namespace: "project:acme", key: "core-thesis" }) memory_read({ namespace: "project:acme", key: "product-scope" }) // readBeforeAnswer: true — then drafts a headline grounded in what it read

The contradiction nobody handled

User turn: "…we're dropping the desktop app. CLI-first is the bet now." The store already holds a record (acme/product-scope) saying the product ships with a desktop app. The correct move is to update that record. Instead, every run wrote a fresh one under a new key:

memory_write({ namespace: "acme", key: "project-direction", // ← brand-new key, not the existing product-scope content: "Team decided to drop the desktop app. The data porting tool is now CLI-first…" }) // the old "ships with a desktop app" record is left untouched → two records now disagree

A genuine decision — filed well everywhere

User turn: "all CLI errors go through the new AppError type … the convention from now on." This is real, durable, shared — and it got filed cleanly in every condition (wording didn't matter here). Under context_ (Sonnet):

context_write({ key: "convention-error-handling", type: "project", tags: ["conventions","errors","cli"], content: "## Error Handling Convention\n\nAll CLI errors must go through the `AppError` type with an associated error code. This applies to every module…" })
